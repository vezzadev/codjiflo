# Questions for Review - 2025-12-31

## Spec Review: models.md vs. Implementation

### 1. Comment Thread Status Model Mismatch

**Observation**: The spec defines `CommentThreadStatus` with 6 values (Active, Pending, Resolved, WontFix, Closed, ByDesign), but the implementation only has a boolean `isResolved` field on `ReviewThread`.

**Question**: Should we implement the full status model from the spec, or should the spec be simplified to match the current GitHub-based implementation?

**Impact**: The richer status model would provide better workflow management but requires UI changes and backend support.

---

### 2. Missing Feedback System

**Observation**: The spec describes `Like` and `CommentFeedback` (NotUseful, Useful, DontUnderstand) systems, but these are not implemented in the code.

**Question**: Are these features planned for a future milestone, or should they be removed from the models.md spec to reflect the current scope?

**Impact**: This affects the Comment interface definition and UI components.

---

### 3. CodeReview vs Review Entity

**Observation**: The spec uses `CodeReview` as the main aggregate root with extensive properties (version, audit log, custom data, etc.), while the implementation uses a simpler `Review` interface focused on GitHub PR data.

**Question**: Is the `CodeReview` model intended as a future abstraction layer that wraps platform-specific entities like GitHub PRs and Azure DevOps PRs?

**Context**: The spec mentions backend abstraction for GitHub/Azure DevOps/GitLab support.

---

### 4. Missing Entities: CodePackage, Layer, WorkItem

**Observation**: The spec defines `CodePackage`, `Layer`, and `WorkItem` entities that are not present in the implementation.

**Question**: 
- Are these Azure DevOps-specific concepts that will be added later?
- Should the spec clarify which models are GitHub-specific vs. platform-agnostic?

**Impact**: Affects the completeness of the data model documentation.

---

### 5. Comment Threading Implementation

**Observation**: The spec describes a hierarchical comment structure with `parentId` and recursive `children: Comment[]`, but the implementation groups comments into threads using `inReplyTo` without building the tree structure.

**Question**: Is the flat thread structure (all comments in a list) intentional for simplicity, or should we implement the full tree traversal as specified?

**Current behavior**: `ReviewThread` contains a flat array of comments, sorted by creation date.

---

### 6. FileRegion Character Positions

**Observation**: The spec defines `FileRegion` with character-level precision (`startCharPos`, `endCharPos`), which aligns with the README's claim of "Character-level precision" for comments. However, the current implementation only tracks line-level positions.

**Question**: Is character-level precision a planned feature for a future milestone, or is line-level sufficient for the initial release?

**Impact**: Affects comment positioning and the "region spanning" feature mentioned in the README.

---

### 7. Validation Rules Documentation

**Observation**: The spec includes validation rules (e.g., "Comment.id >= 0", "text: Required, non-empty"), but these aren't enforced in TypeScript types (which allow nullable/undefined values in many places).

**Question**: Should we:
- Add runtime validation using a library like Zod?
- Add JSDoc comments to clarify required fields?
- Update the spec to match the actual TypeScript type system behavior?

---

### 8. Missing Platform Abstraction Layer

**Observation**: The spec describes `IReviewBackend` and `IFileBackend` interfaces, which exist in `src/api/types.ts`, but many of the models in the spec (CodePackage, Layer, etc.) don't have corresponding backend interfaces.
